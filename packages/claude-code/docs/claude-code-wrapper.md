## claude-code Wrapper Chain

> **Last verified:** 2026-04-12 (commit 0f8ad25). If you touch
> `overlays/claude-code.nix`, the buddy wrapper in
> `packages/claude-code/lib/mkClaude.nix`, or the HM plugin
> wrapper integration and this fragment isn't updated in the same
> commit, stop and fix it.

Claude Code ships as a **pre-built compiled binary** (a Bun
single-exec). The base package (`overlays/claude-code.nix`)
installs it directly as `$out/bin/claude`. When buddy is enabled,
a thin wrapper is layered on top at the HM module level.

### The chain (no buddy)

```
claude             (HM plugin wrapper, from nixpkgs claude-code module)
  → $out/bin/claude  (pre-built binary from overlays/claude-code.nix)
```

1. **HM plugin wrapper** (`$out/bin/claude`): a short bash script
   produced by nixpkgs' `programs.claude-code` module. It execs
   the underlying binary with a `--plugin-dir <store-path>`
   argument pointing at a generated plugin store path.
2. **Pre-built binary**: the compiled Bun single-exec fetched from
   Anthropic's GPG-signed manifest. Bun is embedded in the
   binary — no external Bun dependency needed at runtime.

### The chain (with buddy)

When `ai.claude.buddy.enable = true`, `mkClaude.nix` replaces
`programs.claude-code.package` with a `symlinkJoin` that layers
a buddy-dispatching wrapper script on top:

```
claude             (HM plugin wrapper, from nixpkgs claude-code module)
  → $out/bin/claude  (buddy wrapper, from writeShellScriptBin in mkClaude.nix)
    → $STATE_DIR/claude  (patched binary copy, if activation has run)
    → cfg.package/bin/claude  (store binary fallback)
```

1. **HM plugin wrapper**: same as above — `--plugin-dir` injection.
2. **Buddy wrapper** (`$out/bin/claude` from the `symlinkJoin`):
   a bash script generated by `writeShellScriptBin` inside
   `mkClaude.nix` (only when buddy is enabled). Checks for a
   patched binary at `$XDG_STATE_HOME/claude-code-buddy/claude`.
3. **Patched binary or store fallback**: the wrapper execs the
   patched copy if it exists (`-x` check), otherwise falls back
   to the unmodified store binary.

### Why no separate Bun wrapper

Claude Code migrated from a JavaScript `cli.js` entry point to a
pre-compiled Bun single-exec binary. The binary embeds the Bun
runtime, so there is no need for an external Bun wrapper. The old
two-wrapper chain (`HM plugin → .claude-wrapped → bun run cli.js`)
no longer exists.

### Hash consistency

The any-buddy worker still runs under Bun (via `${pkgs.bun}/bin/bun`
in the activation script) to compute the salt using wyhash. The
pre-built binary also uses wyhash because it IS a compiled Bun
executable. Hash function parity between the worker and the
runtime is guaranteed by the binary format — no wrapper needed.

### The base package

`overlays/claude-code.nix` builds a `stdenv.mkDerivation` that
fetches the platform-specific pre-built binary from Anthropic's
manifest and installs it as `$out/bin/claude`. Per-platform sources
are tracked in `overlays/claude-code-sources.json`, managed by
the package's `updateScript`. No `symlinkJoin`, no
`passthru.baseClaudeCode` — the overlay output is the raw binary
package.

The buddy activation script in `mkClaude.nix` references the
store binary directly via `${cfg.package}/bin/claude` and copies
it to the writable state directory for patching.

### Binary resolution in the buddy wrapper

The buddy wrapper has two code paths:

```bash
BUDDY_BIN="${XDG_STATE_HOME:-$HOME/.local/state}/claude-code-buddy/claude"
if [ -x "$BUDDY_BIN" ]; then
  exec "$BUDDY_BIN" "$@"          # patched binary copy
else
  exec ${cfg.package}/bin/claude "$@"  # store binary fallback
fi
```

When the buddy activation script has run at least once, a writable
binary exists at `$XDG_STATE_HOME/claude-code-buddy/claude` with
the salt patched into it (3 occurrences of the `friend-2026-401`
marker replaced). The wrapper picks that up. Before first
activation (or if the state dir is wiped), the wrapper falls back
to the store binary, which runs claude-code normally without the
custom buddy. No breakage either way — the "gracefully degrades"
design is preserved.

### Symptoms and what they mean

- **`claude` launches but the buddy is wrong**: activation script
  probably didn't run (check `$XDG_STATE_HOME/claude-code-buddy/fingerprint`
  exists) OR the salt the worker computed doesn't match the
  runtime's hash function (both should use wyhash via Bun)
- **`cat $(which claude)` shows the HM plugin wrapper, not our
  buddy wrapper**: expected. The HM plugin wrapper execs the
  underlying package's `claude` binary. When buddy is enabled,
  that binary is actually the buddy wrapper script from the
  `symlinkJoin`. Inspect `bin/claude` in the symlinkJoin store
  path to see the buddy wrapper source.
