## Research + Fix: devenv auto-regeneration

### Root cause investigation

The `.claude/rules/*.md`, `.github/instructions/*.md`, `.kiro/steering/*.md`,
`CLAUDE.md`, and `AGENTS.md` files were NOT managed by devenv `files.*`.
Confirmed by checking `.devenv/state/files.json` — zero entries for rules,
instructions, or steering files. Only skills, MCP configs, settings, and
pre-commit config were managed.

These files were produced by explicit devenv tasks
(`generate:instructions:*`) that run `nix build .#instructions-claude`
and `cp` the output. The `ai.instructions` option in the devenv module
was empty — the stacked-workflows devenv module is a no-op stub (only the
HM module contributes instructions).

The nix build path works correctly — `nix build .#instructions-claude`
rebuilds when fragment sources change. But devenv tasks cached the old
store path. Clearing eval cache (`nix-eval-cache.db`) and running
`devenv update` didn't help because the instruction files aren't
tracked by devenv's eval cache.

### Options considered

- **Option A (rejected):** Unconditional `devenv tasks run generate:instructions` in `enterShell`. Adds 2-5s to every shell entry.
- **Option B (implemented):** Wire `dev/generate.nix` into `files.*` directly. Content computed at eval time, materialized as store-backed files.
- **Option C (backup):** Fingerprint check in `enterShell` — hash fragment sources, regen only on change. Sub-second on cache hit.

### Fix implemented (Option B)

Imported `dev/generate.nix` in `devenv.nix` (same import as `flake.nix`,
single source of truth). The `gen` output feeds `files.*` entries that
materialize instruction files on every shell entry. Zero DRY violation.

Added `coding-standards` overlay to devenv (was only in flake before)
because `gen` needs `pkgs.coding-standards.passthru.fragments`.

**Verified:** `.claude/rules/nix-standards.md` shows updated content
with fresh timestamps after `devenv shell` entry. All ecosystem files
materialized correctly.

**Residual:** Files are written but don't appear in devenv's
`files.json` managed files list. Functionally correct — the files exist
with current content and correct permissions. The tracking gap is
cosmetic and likely a devenv version behavior.

### Interim design: "Stage, Validate, Push" (superseded)

Three-phase workflow: Phase 1 pushes to staging ref, Phase 2 validates
on both platforms, Phase 3 fast-forwards branch. Failure mode: entire
batch held back. Acceptable but user wanted per-dependency granularity.

### Final design: Renovate-style per-dependency PRs

After user feedback, redesigned to match Renovate's model:

1. **Phase 1 (ubuntu):** Run ninja pipeline normally (cherry-picks to
   local branch for build validation)
2. **Phase 2 (ubuntu):** For each update commit between base and HEAD:
   - Create branch `update/<dep-name>` from base
   - Cherry-pick the single commit
   - Force-push (updates existing branch on re-run)
   - Create or update PR against the working branch
3. **ci.yml validates:** PRs trigger ci.yml which builds on both
   platforms (matrix: linux + darwin)
4. **Merge when green:** PRs that pass both platforms can be merged

**Key details:**

- Dep name extracted from commit subject: `chore: update input <name>`
  or `chore(overlays): update <name>` → `update/<name>`
- Non-update commits (treefmt, etc.) are skipped
- Uses `gh pr create` / `gh pr list` for PR management
- Triggers: push to branch + schedule + manual dispatch
- Self-triggering prevention: `github.actor != 'github-actions[bot]'`

**One script change:** `MERGE_LOCK` in `update-common.sh` now respects
env override for CI where `/run/user/$UID/` may not exist.

## Research: Nix doc comment tooling (full)

Also saved to memory at `reference_nix_doc_tooling.md`.

**User's goal:** Extract doc comments from `.nix` source files and feed them into the fragment pipeline as first-class fragments (instead of separate markdown files). Single source of truth — the code IS the documentation.

### RFC 145 doc-comment format (the standard)

```nix
{
  /** This function adds two numbers
    # Type
    add :: Number -> Number -> Number
  */
  add = a: b: a + b;
}
```

`/**` (double asterisk) distinguishes doc-comments from regular `/* */`. Content is CommonMark. Accepted RFC, partially implemented:

- `nix repl :doc` renders them (Nix 2.24+, July 2024)
- nixd LSP shows them on hover
- nixdoc CLI extracts them to markdown

### Tools

**nixdoc** (nix-community/nixdoc): Rust CLI using `rnix` parser. Extracts `/**` comments from `.nix` files → CommonMark markdown. THE tool used by nixpkgs to generate lib reference docs. Works on custom libraries. Invocation:

```bash
nixdoc -c "$name" -d "lib.$name: $description" -f "lib/$name.nix" > "$out/$name.md"
```

**Limitation:** Expects flat attribute sets. Cannot follow dynamic attr generation or deeply nested structures. Our overlays are functions `{ inputs, final, ... }: let ... in derivation` — nixdoc can't parse these directly.

**nix-doc** (lf-/nix-doc): Interactive search + REPL plugin. Three components:

1. CLI search (`nix-doc search <regex> [dir]`) — grep for functions with docs
2. REPL plugin (`builtins.doc f`) — show docs for any lambda
3. Tags generator — vim ctags for Nix files

Complements nixdoc (interactive reading vs static generation).

**docnix** (nix-community/docnix): Ambitious unified doc gen project by @hsjobeki. Has `codemod` for auto-migrating old comments to RFC 145. Includes custom builtins (`unsafeGetAttrDoc`, `unsafeGetLambdaDoc`). NOT production-ready.

**nixd LSP** (nix-community/nixd): Shows RFC 145 doc-comments on hover. Uses real Nix evaluator (not rnix). We already have this in our devenv (nixd in packages).

### How nixpkgs does it

Path: source `/**` comments → `nixdoc` CLI → CommonMark → nixos-render-docs → HTML manual.

The `lib-function-docs.nix` in nixpkgs loops over `libsets = [{name, description}]`, runs `nixdoc` per file, includes output in the manual. `locations.json` maps functions to GitHub source links.

### Extraction approach for our fragment pipeline

To use doc comments as fragment sources:

1. Add `/**` doc-comments to overlay `.nix` files, lib functions, etc.
2. Extract at build time via one of:
   - **nixdoc** on files restructured as flat attrsets (wrapper file)
   - **Custom Nix-native extractor** using `builtins.readFile` + string parsing (no external tool, works on any file structure)
   - **docnix** when it stabilizes
3. Feed extracted markdown into `mkFragment` with `source` pointing at the `.nix` file
4. `compose` and transforms work as normal — the extracted doc becomes a fragment like any other

**Challenge:** Our overlays aren't flat attrsets. Options:

- Write a thin wrapper `.nix` that re-exports the overlay's doc-worthy attributes as an attrset for nixdoc
- Write a custom Nix string parser (`builtins.match` on `readFile` output) to extract `/**` blocks
- Use `nix eval` with `builtins.unsafeGetAttrDoc` (docnix approach, not stable)

### NixOS module option documentation

Module options use `mkOption { description = "..."; }`. Rendered by:

- `nixosOptionsDoc` → `options.json` → NuschtOS Search (we already do this)
- `nixos-render-docs` → HTML manual (nixpkgs only)

We already have this working via `lib/options-doc.nix` + NuschtOS `mkMultiSearch`. The `description` field is the highest-ROI documentation surface for modules.

1. **Phase 1 (ubuntu):** Run ninja pipeline normally (cherry-picks to local branch)

**Key advantage:** Each dependency is independently validated on both
platforms before merging. Failed darwin builds only hold back that
